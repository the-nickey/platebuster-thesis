# streamlit_cloud_eval — бутстрэп-сервис

Отдельный Streamlit-сервис под расчёт 95 % доверительных интервалов и парных
тестов значимости для шести моделей-кандидатов работы. Развёртывается
независимо от production-сборки `streamlit_cloud/`.

## Архитектура

Per-image predictions считаются **локально** в режиме `torch.set_num_threads(1)`
(эмуляция 1 vCPU Streamlit Cloud), сохраняются в JSONL и кладутся в
`data/predictions/`. Бутстрэп считается прямо в облаке — он быстрый, не зависит
от модели и не требует тяжёлых зависимостей (только numpy).

```
streamlit_cloud_eval/
├── app.py                 # Streamlit UI
├── bootstrap.py           # библиотека: precompute + single/paired bootstrap
├── requirements.txt       # streamlit + numpy + pandas (без torch)
├── data/
│   ├── predictions/       # <model>/<region>.jsonl — на вход бутстрэпу
│   └── samples/           # 20 фоток/регион для latency probe (опционально)
└── README.md
```

## Запуск локально

```bash
source ../.venv/bin/activate
pip install -r requirements.txt
streamlit run app.py
```

## Деплой в Streamlit Community Cloud

1. Сложить `data/predictions/<model>/<region>.jsonl` в репу (общий размер
   обычно 10–30 МБ — LFS не нужен).
2. Запушить отдельную репу с этой папкой как корневой.
3. На share.streamlit.io указать `app.py` как entry-point.
4. Зависимости подхватятся из `requirements.txt`.

## Что внутри

**Tab «По моделям»**: для каждой пары (модель × регион) считается mAP@50,
mAP@50:95, mean keypoint error, PCK@0.05 с 95 % перцентильным ДИ через
бутстрэп с возвращением.

**Tab «Парные сравнения»**: для пары моделей (A, B) на каждой итерации
ресэмплируются одни и те же image_id, считается `Δ = metric_A − metric_B`.
ДИ разности и эмпирический двусторонний p-value. Вердикт:
- ДИ ниже нуля — A значимо хуже B
- ДИ выше нуля — A значимо лучше B
- ДИ накрывает 0 — различия незначимы (это и есть «функциональная
  эквивалентность» в строгой формулировке)

**Tab «Экспорт»**: готовые Markdown-таблицы для вставки в работу (замена
таблицы 13 + новая таблица парных тестов).

## Оптимизация

Critical-path оптимизация — precompute. Для каждой картинки один раз делается
матчинг pred↔gt по 10 уровням IoU и сохраняются массивы `(scores, tp, n_gt)`.
На бутстрэп-итерации мы только `np.concatenate` уже посчитанных массивов,
сортируем по score, считаем PR-curve. Это в 100+ раз быстрее, чем
переcчитывать матчинг на каждой итерации.

## Параметры

- `B` — число бутстрэп-итераций (по умолчанию 2000)
- `seed` — для воспроизводимости
- `α` — уровень значимости (0.05 → 95 % ДИ по умолчанию)
